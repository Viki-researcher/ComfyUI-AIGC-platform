# 用户、项目、生成日志、仪表盘 — 归属逻辑与潜在漏洞分析

> 本文档梳理数据生成平台中用户、项目卡片、生成日志、仪表盘之间的数据流与归属关系，并分析可能导致「跨用户/跨项目」记录错乱的潜在漏洞。
> 编写目的：为排查「不同用户生成记录归位到别的用户」「生成日志与输出目录混入别的项目图像」等 Bug 提供逻辑依据。
>
> **实际现象**：预览中未见其他项目图像，但**生成日志**与 **runtime/output**（统一输出目录）会混入其他项目的生成图像。

---

## 1. 数据模型关系概览

```
┌─────────────┐       owner_user_id        ┌─────────────┐
│    User     │◀────────────────────────────│   Project   │
│  (用户表)   │                              │  (项目表)   │
└─────────────┘                              └──────┬──────┘
       ▲                                            │
       │ user_id                                     │ project_id
       │                                            │
       │         ┌──────────────────────────────────┘
       │         │
       │         ▼
       │   ┌──────────────────┐         project_id        ┌─────────────────┐
       └───│ ComfyUIService    │◀─────────────────────────│  GenerationLog  │
           │ (ComfyUI 服务)    │                          │  (生成日志)      │
           │ - user_id         │─────────────────────────▶│ - user_id        │
           │ - project_id      │                          │ - project_id     │
           │ - comfy_url       │                          │ - prompt_id      │
           │ - base_dir        │                          │ - image_count    │
           └──────────────────┘                          └─────────────────┘
                    │
                    │ 输出目录映射
                    ▼
           ┌──────────────────────────────────────────────────────────────┐
           │  OUTPUT_BASE_DIR / p{project_id} / {YYYYMMDD} /                 │
           │  （v0.4+ 已改为 project_id，避免项目名冲突）                  │
           └──────────────────────────────────────────────────────────────┘
```

### 1.1 核心表结构

| 表 | 关键字段 | 说明 |
|----|----------|------|
| `projects` | `id`, `name`, `code`, `owner_user_id` | 项目归属 `owner_user_id`，`code` 唯一，`name` 可重复 |
| `comfyui_services` | `user_id`, `project_id`, `comfy_url`, `base_dir` | 每个项目最多一个 ComfyUI 实例，`unique_together=(user_id, project_id)` |
| `generation_logs` | `user_id`, `project_id`, `prompt_id`, `image_count` | `unique_together=(project_id, prompt_id)`，`prompt_id` 在 sync 时按全局唯一处理 |

## 2. 生成日志的创建路径

### 2.1 路径一：History Sync（主路径）

**位置**：`app/services/comfyui_history_sync.py` → `sync_once()`

**流程**：
1. 查询所有 `status="online"` 的 `ComfyUIService`
2. 按 `(has_callback(project_id), project_id)` 排序，有回调缓存的项目优先
3. 对每个服务 `s`：
   - 请求 `GET {s.comfy_url}/history` 获取该 ComfyUI 实例的执行历史
   - 对每个 `prompt_id`：若 `GenerationLog.filter(prompt_id=...).exists()` 则跳过
   - 否则创建 `GenerationLog`，使用 `user_id=s.user_id`, `project_id=s.project_id`
   - 将输出图片复制到 `{OUTPUT_BASE_DIR}/{safe_name}/{YYYYMMDD}/`，`safe_name` 来自 `project.name`

**归属来源**：`user_id`、`project_id` 均来自 `ComfyUIService`，即「谁启动的 ComfyUI、属于哪个项目」。

### 2.2 路径二：ComfyUI 回调（辅助路径）

**位置**：`app/api/internal/comfy.py` → `comfy_callback()`

**流程**：
1. ComfyUI 自定义节点（如 Google-Gemini-Parallel-MultiImage）在生成完成后，读取环境变量 `PLATFORM_PROJECT_ID`，向平台 `POST /api/internal/comfy/callback` 发送 `project_id`、`image_count`、`status` 等
2. 平台将回调数据缓存到 `_callback_cache[project_id]`，**不直接写库**
3. History Sync 在消费 history 时，若 `has_callback(project_id)` 为真，则优先处理该项目，并用回调中的 `image_count` 覆盖从 history 解析的数量

**归属来源**：`project_id` 来自 ComfyUI 进程启动时注入的 `PLATFORM_PROJECT_ID`（与 `ComfyUIService.project_id` 一致）。

### 2.3 路径三：手动写入（兼容接口）

**位置**：`app/api/compat/logs.py` → `POST /api/logs`

**流程**：
1. 请求体包含 `project_id`，`user_id` 来自 `CTX_USER_ID.get()`（当前登录用户）
2. 直接创建 `GenerationLog`

**归属来源**：`user_id` 为当前用户，`project_id` 由调用方传入，**无项目所有权校验**，存在被滥用的可能。

## 3. 项目与输出目录的映射关系

### 3.1 目录规则（v0.4+ 已修复）

**新规则**（唯一）：
```
OUTPUT_BASE_DIR / p{project_id} / {YYYYMMDD} / {safe_name}_{timestamp}_{stem}.png
```

- 目录名使用 `p{project_id}`，确保每个项目有唯一输出目录
- 文件名仍使用 `safe_name`（项目名）便于识别

**旧规则**（已弃用，仅作向后兼容）：
```
OUTPUT_BASE_DIR / {safe_name} / {YYYYMMDD} / ...
```
其中 `safe_name = sanitize(project.name)`，存在项目名冲突风险。

### 3.2 使用该映射的接口

| 场景 | 文件 | 逻辑 |
|------|------|------|
| History Sync 复制图片 | `comfyui_history_sync._organize_output_images` | 复制到 `p{project_id}/{date}/` |
| 项目预览图像列表 | `projects.list_project_images` | 优先 `p{project_id}`，不存在则回退旧路径 |
| 项目浏览（HTML） | `projects.browse_project_images` | 同上 |

## 4. 仪表盘与统计

**位置**：`app/api/compat/dashboard.py`

- **今日生成量**：`GenerationLog.filter(created_at__gte=today_start, status="成功").annotate(Sum("image_count"))`
- **活跃项目数**：`Project.all().count()`
- **平台用户数**：`User.all().count()`
- **生成成功率**：成功数 / 总数

**无用户/项目过滤**，为全局统计。

## 5. 潜在漏洞与根因分析

### 5.1 【高】项目名称冲突导致输出目录混用（已修复）

**现象**：统一输出目录 `runtime/output` 与生成日志中，出现其他项目的生成图像。

**根因**：
- `projects` 表仅对 `code` 做唯一约束，**`name` 可重复**
- 原逻辑：输出目录使用 `safe_name = sanitize(project.name)`，与 `project_id` 无关
- 若项目 A（id=1, name="项目A"）与项目 B（id=2, name="项目A"）存在，两者 `safe_name` 均为 `"项目A"`
- 所有写入与读取都指向同一目录 `OUTPUT_BASE_DIR/项目A/`，导致跨项目混图

**修复**（v0.4+）：输出目录改为使用 `p{project_id}`，确保每个项目有唯一目录。预览/浏览接口保留对旧路径（项目名）的向后兼容。

---

### 5.2 【中】多服务共享同一 comfy_url 导致归属错乱（端口复用 + 心跳延迟）

**端口分配机制**（`comfyui_manager._pick_free_port`）：
- 在 `COMFYUI_PORT_RANGE`（默认 8200–8299）内顺序尝试 `bind()`，第一个成功绑定的端口即被使用
- **无持久化端口注册表**：端口仅记录在 `ComfyUIService.port` 和 `comfy_url` 中
- 进程退出后，操作系统释放端口，下次 `start_instance` 可能再次分配到同一端口

**comfy_url 为何会重复**：
1. 项目 A 的 ComfyUI 在 8200 运行，`ComfyUIService(project_id=A, port=8200, status=online)`
2. 进程崩溃或被外部 kill，端口 8200 被释放
3. **心跳尚未执行**（默认 30 秒间隔）：项目 A 的记录仍为 `status=online`
4. 项目 B 启动，`_pick_free_port` 分配到 8200，创建 `ComfyUIService(project_id=B, port=8200, status=online)`
5. 此时存在**两条** `status=online` 且 `comfy_url` 相同的记录

**History Sync 的归属竞争**：
- Sync 遍历所有 `status=online` 的服务，对每个服务请求 `GET {comfy_url}/history`
- 两条记录指向同一 URL，会拉取同一份 history
- 对每个 `prompt_id`，**先处理到的服务**会创建 `GenerationLog` 并复制图片到其项目目录
- 排序规则：`(has_callback, project_id)`，无回调时按 `project_id` 升序，**project_id 较小的可能错误地“抢到”归属**

**结论**：端口复用 + 心跳延迟会导致多服务共享同一 comfy_url，进而造成生成日志与输出目录归属错乱。

---

### 5.3 【低】ComfyUIService.user_id 与项目 owner 不一致

**流程**：
- `open_comfy` 校验 `project.owner_user_id == user_id`，仅项目所有者可启动
- `restart_comfyui_service` 在更新已有记录时**不更新 `user_id`**，沿用创建时的 `user_id`

**风险**：
- 若历史数据中存在 `ComfyUIService.user_id != project.owner_user_id`（如迁移、手动改库），则 History Sync 会以错误的 `user_id` 写入 `GenerationLog`

**结论**：正常业务流程下不会发生，需关注数据迁移与人工修改。

---

### 5.4 【低】手动写日志接口无项目归属校验

**位置**：`POST /api/logs`（`GenerationLogCreate`）

- `user_id` 固定为当前登录用户
- `project_id` 由请求体传入，**未校验当前用户是否为项目 owner 或是否有权限**

**风险**：恶意或错误调用可将任意 `project_id` 的生成记录挂在当前用户下，造成「用户 A 的日志中出现用户 B 的项目记录」或反之。

## 6. 数据流与归属关系总结

| 环节 | 用户归属 | 项目归属 | 可能出错点 |
|------|----------|----------|------------|
| ComfyUI 启动 | `ComfyUIService.user_id` = 调用 `open_comfy` 的当前用户（须为 owner） | `ComfyUIService.project_id` = 项目 ID | 权限校验失败、历史脏数据 |
| History Sync 写库 | 来自 `ComfyUIService.user_id` | 来自 `ComfyUIService.project_id` | 多服务共享同一 URL、回调缺失 |
| 输出目录与预览 | 无直接用户字段 | 由 `project.name` → `safe_name` 决定 | **项目名冲突** |
| 仪表盘统计 | 无过滤 | 无过滤 | 依赖 `GenerationLog` 正确性 |
| 生成日志查询 | 可传 `user_id`/`project_id` 筛选 | 同上 | 依赖 `GenerationLog` 正确性 |

## 7. 修复与建议

### 7.1 已实现：输出目录使用 project_id（v0.4+）

- 统一输出路径：`{OUTPUT_BASE_DIR}/p{project_id}/{YYYYMMDD}/`
- 预览/浏览接口：优先使用 `p{project_id}`，若不存在则回退到旧路径（项目名）以兼容历史数据

### 7.2 待考虑：多服务共享 URL 防护

- 在 History Sync 中检测 `comfy_url` 重复，对重复 URL 仅由「有回调」或「最早创建」的服务处理
- 或在停止 ComfyUI 时立即将对应 `ComfyUIService` 设为 `status=offline`，缩短心跳检测窗口

### 7.3 待考虑：手动写日志权限

- 对 `POST /api/logs` 增加项目归属校验：仅允许为当前用户为 owner 的项目写入

### 7.4 数据一致性检查

- 定期校验 `ComfyUIService.user_id == Project.owner_user_id`，以及 `GenerationLog` 与 `ComfyUIService` 的 `project_id` 对应关系
